1. 项目概览

1.1 项目定位

Nexus 是一个有向图驱动的子进程编排引擎。用户（或 AI Agent，如 Claude Code）以 JSON 格式定义 DAG 工作流，引擎按定义机械执行，Dashboard 提供实时可视化监控。项目采用 Rust 编写，通过 MCP (Model Context Protocol) 与 Claude Code 深度集成。

核心价值主张：将 AI Agent 的"探索式"工作与确定性工作流引擎结合。Claude 定义 JSON → MCP 触发 → Dashboard 实时监视 → 出问题介入。适合已知流程、需要可靠重复执行的场景。

2.2 核心设计决策

🔷 h_e + g_e 正交分解（边模型的数学基础）

每条边是两个正交函数的组合：

  • h_e（分支匹配函数）：纯函数，判断事件是否匹配 —— event_type + exit_reason + threshold。无记忆、无 triggered 状态。
  • g_e（策略聚合函数）：Any (∨) 单源触发 / All (∧) 全源汇聚。All/Any 是完备的二元组合。
评价：这是项目中最优雅的设计。通过将边建模为纯函数而非带状态的状态机，自然地支持了有向环（directed cycle），而不需要特殊的"触发后重置"逻辑。每次事件独立评估所有匹配边，是教科书级别的不含时（time-independent）设计。

🔷 调度与数据分离

edges 数组控制执行顺序（控制流），dataflows 数组控制数据流向（数据流）。二者独立声明，互不耦合。DataRouter 维护预建索引，O(input_count) 时间内完成数据注入。

🔷 路由策略（Route Policy）

通过 route_policy.max_runs 实现环路自动终止。节点无需感知自己在循环中执行了多少次 —— 当 run_count >= max 时，引擎自动将 exit_reason 覆盖为 then_route，使环路优雅退出。这是将控制逻辑从节点中抽离的典范。

🔷 LLM 原生集成

通过 llm_node.py Python wrapper 实现任意 CLI 适配。Rust 侧负责模板渲染和上下文注入（stdin），Python 侧负责跨平台 CLI 解析、子进程管理和输出解析。设计上支持 Claude、OpenCode 以及任何接受命令行参数的 LLM CLI。

🔷 模板引擎

支持 {{metadata.run_count}}、{{metadata.timed_out}}、{{datarouter.<alias>.route}}、{{datarouter.<alias>.content}}。Shell 模式下自动转义防注入。Validator 在编译期检查所有模板引用的合法性。

2.3 执行模型

引擎采用经典的 事件循环 + 信号量并发控制 模型：

  1. 初始化：GraphDef 构建 → Scheduler + DataRouter 初始化 → Semaphore 创建
  2. 种子注入：所有 entry nodes 作为 NodeReady 事件入队
  3. 事件循环：
    • NodeReady：获取信号量许可 → 提取节点上下文 → 渲染模板 → 生成 NodeContext → spawn 子进程（不阻塞循环）
    • NodeCompleted：收集 stdout/stderr → 解析 NodeOutput JSON → 存储到 DataRouter → 调用 Scheduler::handle_event → 可能触发下游 Ready
  4. 收敛检测：Scheduler::is_converged()（所有节点处于终态 + ready_queue 为空）
  5. Watchdog (10s)：防止事件丢失导致的死等；3 次连续 timeout 后标记剩余节点 Skipped 并退出
  6. 外部取消：AtomicBool 标记，下一轮循环时检测并收敛

    不是遇到第一个错误就返回，而是收集所有验证错误后一并报告。对用户体验友好。

🟡 待改进项

  1. 模板渲染代码重复（严重度：中）

    LlmExecutor::render_inputs() 和 SubprocessExecutor::render_template() 包含几乎相同的模板解析逻辑（{{metadata.*}} / {{datarouter.*.*}}）。建议提取到共享的 TemplateEngine 结构体。

  2. HTTP Provider 为占位符（严重度：低）

    ProviderDef::Http 已定义但标记为 "reserved for future use"，无实际执行逻辑。

  3. Dashboard 状态为内存存储（严重度：中）

    虽然 SQLite 持久化了 workflow 和 run 元数据，但 WebSocket room 状态和 cancel_flags 完全在内存中。Dashboard 重启后会丢失所有进行中的运行。

  4. Shell 注入风险文档化但不完全防护（严重度：中低）

    SubprocessExecutor::new_shell() 的文档明确警告不消毒所有 shell 元字符（反引号、$()、&&），依赖调用方自行确保输入安全。但在 CLI 路径上，工作流 JSON 通常由可信的 AI Agent 生成，实际风险较低。

  5. 缺少集成测试（严重度：中）

    虽然单元测试覆盖很好（168+），但端到端集成测试（完整的 workfow JSON → 引擎执行 → 验证输出）较少。MCP server 有 11 个集成测试，但 engine 和 CLI 的集成测试覆盖有限。

  6. 代码量集中（严重度：低）

    runtime/engine.rs 约 2,245 行，是最长的单文件。随着功能增长，可能需要进一步拆分。

  7. 缺少性能基准测试（严重度：低）

    没有 benchmark 或性能测试。对于编排引擎，了解大规模 DAG（如 100+ 节点）的执行开销是有价值的。

6.2 最值得称道的工程实践

  1. GraphDef 的"单一构造路径" — from_components() 是创建 GraphDef 的唯一入口，强制所有不变量检查。这是 "make illegal states unrepresentable" 的 Rust 哲学体现。
  2. NodeTransfer::evaluate — 将转移函数建模为命名实体而非内联逻辑，使其成为可独立测试的单元。19 个 scheduler 测试 + 7 个 graph_def transfer 测试从不同层级验证同一逻辑。
  3. Watchdog 的三阶段策略 — 10s 无事件 → 检查收敛 → 检查 running_count → 3 次连续超时才标记 Skipped。优雅处理了慢节点 (LLM) 和真正死锁两种场景。
  4. rerun 协议 — 节点通过 {{metadata.timed_out}} 感知上次是否超时，实现自适应重试策略。这是 LLM 场景下的关键设计。
  5. fan_in_pending 的正确重置 — All 策略触发后重置 pending 计数回初始值，支持多轮 fan-in（如 review→fix→review 循环中的 merge 节点）。正确处理了 "All 策略在循环 DAG 中的可重用性"这个非平凡问题。

7. 改进建议

7.1 短期（1-2 周）

  1. 提取共享模板引擎：将 LlmExecutor::render_inputs 和 SubprocessExecutor::render_template 的公共逻辑提取到 TemplateEngine，消除约 60 行重复代码。
  2. 添加 engine 集成测试：至少 3 个端到端测试（链式 JSON → 完整执行 → 验证输出顺序和内容），确保 CLI 路径的正确性。
  3. 补充 runtime/engine.rs 的单元测试：当前 engine 无独立测试，依赖集成测试间接覆盖。建议将 event loop 核心逻辑提取为可测试的纯函数。
  4. 修复 test 计数显示问题：运行 cargo test --list 确认准确的测试总数，更新 README badge。

7.2 中期（1-2 月）

  1. 实现 HTTP Provider：补全 ProviderDef::Http 的执行逻辑，支持通过 HTTP 调用外部服务作为工作流节点。
  2. Dashboard 状态持久化：将运行中的 cancel_flags 和 WS room 状态持久化到 SQLite，支持 Dashboard 重启恢复。
  3. 添加 benchmark：使用 criterion 对大规模 DAG（10/50/100 节点）进行调度性能和吞吐量基准测试。
  4. Shell provider 安全加固：增加更严格的元字符校验或提供 allow_unsafe_chars 配置开关。
  5. runtime/engine.rs 拆分：将 2,245 行的文件按关注点拆分为 event_loop.rs、node_handler.rs、watchdog.rs。

7.3 长期（3-6 月）

  1. 分布式执行：支持跨多台机器的节点分发执行，引入 gRPC 或 NATS 作为通信层。
  2. 工作流版本管理与回滚：在 Dashboard 中支持工作流定义的版本历史和差异对比。
  3. 条件数据流：支持基于运行时值的动态 dataflow 路由（当前 dataflow 是静态声明的）。
  4. 插件系统：允许用户以动态库形式注册自定义 Provider 类型。
  5. WebAssembly 节点：支持 WASM 作为沙箱化节点执行环境，提供更强的安全隔离。

8. 总体评价

Nexus 是一个设计精良、代码质量优秀的 Rust 项目。

它的核心优势在于：

  • 坚实的理论基础：h_e + g_e 的正交分解使边模型兼具简洁性和表达能力
  • 严格的工程纪律：从 lint 配置到不变量检查，从 Validator 到 GraphDef，每一层都有防御性编程的痕迹
  • 务实的集成设计：MCP + Dashboard + CLI 三种接口覆盖了 AI Agent 交互、人工监控和脚本自动化的全部场景
  • 优雅的环路支持：route_policy 将环路终止逻辑从节点中剥离，是控制反转（IoC）的典范

作为一个 v0.1.0 的项目，当前状态令人印象深刻。核心引擎的架构质量已经达到了可投入生产的水平。主要待补足的是集成测试、HTTP provider 实现，以及 Dashboard 的状态持久化。按照上述路线图推进，Nexus 有潜力成为一个在 AI Agent 工作流编排领域独树一帜的工具。
