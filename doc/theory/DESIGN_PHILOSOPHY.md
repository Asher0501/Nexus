# Nexus 设计哲学

> 全局 = 局部信息的迭代闭包。引擎不是控制器，是闭包求值器。

**分类**：theory

---

## 〇、核心命题

**命题（局部闭包定理）：**
对于任意有限有向图 G = (V, E)，若每个顶点 v ∈ V 附带一个局部转移函数 f_v : State_v → 2^V（将 v 的局部状态映射到需要触发的下游顶点集），则全局执行轨迹 T(G) 等价于所有 f_v 在状态空间上的最小不动点。

### 收敛条件（不动点）

当所有 Running 状态的节点执行完毕、就绪队列为空时，所有转移路径达到不动点，引擎停止：

```
is_done = (running_count == 0) && ready_queue.is_empty()
```

引擎不需要知道全局图——它只追踪两个信号：还有节点在跑吗？还有节点排队吗？都为空，则工作流已收敛。

### 命题推论

**推论 1（等价性）：** 全局执行轨迹 T(G) = 局部转移函数 f_v 的迭代闭包。不需要中心化控制器——引擎是闭包求值器，不是流程管理器。

**推论 2（终止性）：** 当所有 Running 状态的节点执行完毕、就绪队列为空时，所有 f_v 达到不动点，引擎收敛。引擎不需要知道全局图——只需要追踪 `running_count == 0 && ready_queue.is_empty()`。

**推论 3（完备性）：** All（∧）和 Any（∨）可以表达任意布尔触发逻辑。不需要第三种策略。

## Exec 模型等价性定理

**问题**：Nexus 采用去中心化的异步事件驱动模型——引擎不扫描全局图，只被动响应节点完成事件，调用已完成节点 v 的局部转移函数 F_v。这个模型和一个假设的"中心化同步调度器"模型会产生相同的路径集和终态吗？

**结论**：会。两者使用完全相同的转移函数，在每一步的状态转移上保持一致。

---

### 一、形式化基础

**转移函数的不含时性**：F_v 的输入变量为系统状态变量——不读系统时钟、不记调用序号、不依赖事件到达时序。F_v 的判定结果仅由当前状态和当前事件决定。

**核心观察**：F_v 是一个定义在状态空间上的马尔可夫转移核。模型的全部状态演化由初始状态和事件序列唯一确定，与"谁来调用 F_v"无关。

---

### 二、前提条件

**P1 — 转移核同一性（Kernel Identity）**
两个模型共享完全相同的转移核：同一组转移函数 {F_v | v ∈ V}，作用于同一个底层结构。两个模型对 F_v 的访问语义一致。

**P2 — 初始状态齐一性（Initial State Alignment）**
两个模型从完全相同的初始配置开始：入口节点集相同、数据路由为空、就绪队列相同。

**P3 — 转移函数的确定性（Determinism of F_v）**
F_v 是纯函数：相同的输入总是产生相同的输出。不依赖随机数、系统时钟或外部 I/O。

---

### 三、模型定义

两类模型运行于同一标架之上，共享完全相同的底层结构。

**模型 D（去中心化·异步）**——Nexus 的实际执行模型：

```
on_completed(v):
  存储 v 的输出到 DataRouter
  遍历所有以 v 为上游的下游边：
    更新汇聚状态
    如果汇聚条件满足：
      从 DataRouter 组装下游输入
      下游入队
```

**模型 C（中心化·同步）**——假设的对照模型：
与 D 的差异仅在于调用源，执行完全相同的算法。

---

### 四、定理与证明

**定理（执行模型等价性）：**
令 Φ 为满足 P1 的底层系统。令模型 D 和 C 从满足 P2 的初始状态启动。假设 F_v 满足 P3。则对于任意次状态转移，两模型在每一步后保持完全一致。

**证明**：对引擎处理的事件个数 k 进行归纳。

*基例 k = 0*：无事件发生，两模型保持初始状态。由 P2 直接得相等。

*归纳假设*：前 k 个事件处理后，两模型状态完全一致。

*归纳步骤*：

**(a) 事件来源的一致性**：Q 非空时，两模型从各自 Q 中出队下一个待执行节点。由归纳假设出队的节点 v 相同。该节点执行完毕后产生相同的输出。

**(b) 状态转移的一致性**：两模型收到 v 完成通知后均调用 on_completed(v)，执行相同的算法。由 P3 保证确定性。每一步分支判定相同，状态更新相同。

**(c) 就绪队列的一致性**：被触发的下游节点集相同、入队顺序相同。

三部分之和：状态完全一致。归纳完成。∎

---

### 五、推论

**推论 1（路径等价性）：** 两模型产生的执行路径完全相同。

**推论 2（收敛等价性）：** 引擎收敛时两模型的终态相同。

**推论 3（确定性）：** 给定相同的图拓扑和初始状态，引擎的执行路径由事件序列唯一确定——不因调用方是谁而改变。

---

### 六、辅助理解

**为什么等价性几乎是不证自明的？**
D 和 C 共享完全相同的转移函数、完全相同的初始状态、完全相同的图定义。如果同一个数学函数在同一个输入下会产生不同输出，那它就不是一个数学函数。

**前提的作用：**

| 前提 | 防止的问题 |
| --- | --- |
| P1 | Builder 给 D 和 C 生成了不同的边定义 |
| P2 | 两模型对入口节点的识别规则不一致 |
| P3 | F_v 中出现了随机数或外部 API 调用 |

**这个定理回答了什么问题：**

- **"去中心化会不会漏掉某些触发？"** 不会。两模型使用完全相同的判定逻辑。
- **"事件到达的乱序会不会影响路径？"** 不会。转移函数 F_v 的输入中不存在时间变量，交换互不相交的 F_v 的调用顺序不影响最终状态。
- **"这个证明是否 trivial？"** 在某种意义上是的——这正是要点。证明的简洁性本身就是架构正确性的信号。

## All / Any 完备性证明

**主张：** 任意布尔逻辑表达式（由 `∧`、`∨` 组合的上游触发条件）都可以用 All 和 Any 两种边在图上等价表达。

**证明思路：** All 对应合取（`∧`），Any 对应析取（`∨`）。通过引入辅助节点，任意嵌套的 `∧` / `∨` 表达式可以展开为两层 All/Any 边的图结构。

**引理 1（All = ∧）：**
一组上游 `{A, B}` 的 All 边等价于"A 完成 ∧ B 完成"。

```
A ──All──→ D   ≡   D 就绪当且仅当 A 完成且 B 完成
B ──All──→ D
```

**引理 2（Any = ∨）：**
一组上游 `{A, B}` 的 Any 边等价于"A 完成 ∨ B 完成"。

```
A ──Any──→ D   ≡   D 就绪当且仅当 A 完成或 B 完成
B ──Any──→ D
```

**定理（完备性）：** 对于任意由 `∧` 和 `∨` 构成的布尔触发表达式 T，存在一个图 G 使得 G 的执行语义等价于 T。

*证明（构造法，对表达式深度归纳）：*

*基例：* 单个变量 A → 以 A 为唯一前驱的边即可。

*归纳步骤：*

**情况 1 — T = T₁ ∧ T₂**（合取）

设 G₁ 是 T₁ 的图实现，以节点 X 输出信号。设 G₂ 是 T₂ 的图实现，以节点 Y 输出信号。构造中间节点 M：

```
X ──All──→ M
Y ──All──→ M
M ──Any──→ D
```

当且仅当 X 和 Y 都产生信号后，M 收到 All 触发，转发给 D。与 T₁ ∧ T₂ 一致。

**情况 2 — T = T₁ ∨ T₂**（析取）

```
X ──Any──→ M
Y ──Any──→ M
M ──Any──→ D
```

X 或 Y 任一产生信号，M 收到 Any 触发，转发给 D。与 T₁ ∨ T₂ 一致。

**推论：** All 和 Any 作为条件组合原语是完备的——任何由 `∧` / `∨` 表达的上游触发逻辑都可以通过有限层级的 All/Any 边和中间节点在图拓扑中等价实现。
