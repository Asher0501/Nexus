# 设计原则与工程洞见

> 从实践中提炼的通用经验。内容与具体项目无关，Nexus 只是这些原则的一个实例。

**分类**：reference

---

## 一、I/O 层

### 管道不是终端

任何进程在 stdout 连接 pipe 时使用块缓冲（4-8KB），连接 TTY 时使用行缓冲。这是 libc 的默认行为（`isatty()`），跨语言、跨平台。表象可能不同（输出不实时、不逐行），根因相同。

解法三条路径：

| 路径 | 做法 | 代价 |
|------|------|------|
| 改子进程缓冲 | `bufsize=1`，环境变量强制行缓冲 | 依赖子进程支持 |
| pseudo-terminal | `pty.spawn()` 模拟 TTY | 平台相关，Windows 需 ConPTY |
| consumer 端非阻塞 read | `read1()` 逐字节读取 | 需要自行分行 |

### 命令行不是数据通道

所有 OS 的命令行参数都有长度限制和字符限制。非受控内容（用户输入、LLM 生成文本）放入命令行参数是结构性错误——长度超限静默截断，换行符破坏参数解析。**stdin 才是数据通道。**

### 进程边界是编码断层

跨进程通信可能引入 surrogate（U+D800-U+DFFF）。surrogate 在单进程内可以无损传递（多数语言的字符串可以存），但编码为 UTF-8 时必炸。**所有进程边界的出口处都必须 sanitize。**

## 二、解析层

### Parser 必须比协议宽容

当输出的生产者是非确定性的（LLM、人类、第三方系统），parser 不能只接受"正确的"格式。协议定义的是规范，parser 要接受规范的超集——缺字段、多余包装、噪声混入都应容错。

核心手段是**逐行 `json.loads()` 而非整块 regex**。行的边界比模式的边界更容易预测：一行要么是合法 JSON 要么不是，不需要猜哪里开始哪里结束。regex 是 fallback，不是主力。

### 不要在生产者的输出中放消费者的格式模板

当 parser 在整个输出空间搜模式时，模板的 echo（日志、tool result 回显）会被误匹配。**格式模板应该只存在于 parser 不会搜索的地方。**

## 三、设计哲学

### 静态检测 > 运行时崩溃

如果能从输入定义本身判断某个条件必然导致失败，检查就应该放在启动阶段而非执行阶段。不是因为运行时处理不了，而是反馈周期——秒级 vs 分钟级。

判别标准："给定这份输入，能不能在执行前就知道它会失败？"

### 引擎修"换了任何场景都会炸"，文档记"知道方法就能绕"

边界筛子是**普适性**，不是严重程度：

```
换了任何业务场景都会触发？ → 引擎
需要特定任务结构才会遇到？ → 文档
```

### 可观测性是信任基础，不是功能

用户看不到系统在做什么 → 用户不相信系统在工作。三项不可协商：状态变化实时反映、中间产物实时可见、连接恢复后主动同步当前态。缺失任何一项，其余功能的质量都会被低估。

## 四、AI 交互

### Attention 的尾部优先效应

LLM 的 attention 机制天然赋予末尾 token 更高的决策权重。"先读文件，再审查，最后输出 JSON"——LLM 会优先满足末尾的格式约束，而非开头的任务描述。

解法不是强调"要先做工作"，而是**把格式约束从工作阶段完全移除**——工作阶段没有格式指令，LLM 无选择只能先工作。格式指令只在工作完成后才出现。

### 工具调用是执行的唯一证据

LLM 把"描述如何做"视为任务完成，因为它确实完成了理解和表达。区分"理解了"和"执行了"的唯一标志是工具调用——prompt 的设计目标不是"让 LLM 知道要做什么"，而是"让 LLM 必须调用工具才能宣称完成"。

### 输出格式的优先级天然高于工作内容

LLM 的奖励模型更偏向满足显式的格式指令（"输出 JSON"）而非隐式的任务指令（"审查代码"）。不是某个模型的问题，是 RLHF 的结构性偏差：格式满足是可量化的即时奖励，任务完成是难以量化的延迟信号。

Phase 分离不是 prompt 技巧——是在对抗这个偏差。

## 五、API 设计

### 接口接受常见错误格式

用户犯错是必然的。接口应该检测并纠正可判定的格式偏差，不是因为"纵容错误"，而是因为这类错误对用户没有信息价值——他本意不是传裸 JSON，只是不知道要包装。纠正后系统正常工作，用户体验平滑。

文档仍然写正确格式。纠正只是消除不必要的错误类别。

## 六、节点编排

### 节点的粒度定义框架的性质

一个框架中"节点"代表什么，决定了它的抽象层次和适用场景：

| 节点 = 完整业务任务 | 节点 = 原子推理操作 |
|-------------------|-------------------|
| 编排多个 Agent 协作 | 构建一个 Agent 的推理流 |
| tool loop 对引擎透明 | tool loop 是图上的边 |
| 静态拓扑 + 收敛保证 | 动态拓扑 + 编程式控制 |

两种选择没有好坏——但一旦选定，工具链、可观测性、错误模型都必须在这个抽象层次上保持一致。

### 双通道：控制信号 ≠ 数据载荷

任何需要节点间传递 "做什么" 和 "怎么做" 的循环都需要两条通道：

| 通道 | 载体 | 内容 | 生命周期 |
|------|------|------|---------|
| 控制 | stdout JSON `route` + `content` | 路由决策 + 一句话摘要 | 单轮（被下轮覆盖） |
| 数据 | 文件系统 | 详细分析 + 审计追踪 | 持久化（跨轮累积） |

不要让路由协议承载数据，也不要用文件替代路由。

## Human-in-the-Loop 中的经验

### 架构原则

**组件生命周期对齐。** Pool 应比任一 wrapper 长。用 `subprocess.Popen` 而非 `import` 启动独立服务，避免绑定到调用者进程树。

**阻塞等待优于轮询，但需要多线程。** `Event.wait()` 零 CPU，但必须配合多线程 server（如 `ThreadingMixIn`），否则一个阻塞请求卡死整个服务。

**子进程的 stdin/stdout/stderr 被父进程接管时，终端 IO 需独立通道。** 引擎接管了 wrapper 的 stdio，wrapper 不能从终端读。解决方案是独立的 consumer 进程/线程。

**Shell/管道编码不一致是跨平台子进程通信的经典陷阱。** Windows 子进程 stdout 默认用系统代码页，非 UTF-8。Python 输出用 `ensure_ascii=True` 或设 `PYTHONIOENCODING=utf-8`。Rust 侧 `serde_json` 自动解码 `\uXXXX`。

**文件路径不依赖 CWD。** 子进程 CWD 取决于父进程启动目录。工作流中传给 LLM 的文件路径用绝对路径。

### 设计模式

**生产者-消费者解耦，UUID 做关联。** 每个生产者 POST UUID 标识，Consumer 写答案时指定 UUID。不共享状态，不依赖顺序。

**自启动服务的健康检查。** 先 `GET /health` 探测，失败再启动，启动后循环等待就绪。幂等——多次调用安全。

**交互面与核心逻辑分离。** ToolBridge 只负责注册问题+等待答案。怎么展示问题、收集答案，由 CLI/Dashboard 各自实现。核心逻辑零依赖交互面。

### LLM 交互

**LLM 自主决定何时问人。** `ask_human` 是普通 Anthropic tool。LLM 自己判断"不确定"→ 调 tool。不需要在 workflow JSON 中预设交互节点。

**Tool description 决定使用率。** 描述必须说清楚场景。太模糊 LLM 永远不调。
