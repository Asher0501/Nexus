# 语言选型：为什么选择 Rust

> 本文档与架构设计、设计哲学正交——它不讨论"引擎怎么设计"，只讨论"为什么用 Rust 而不是其他语言"。
> 本文假设读者不懂任何编程语言，出现的术语都会解释。

---

## 一、先看这个项目需要什么

Nexus 做的事情：**启动各种程序（节点）、等它们做完、根据结果决定下一步、同时管理很多个节点的运行。**

这就像一条工厂流水线，但不是人工流水线——是程序自动控制的。控制这条流水线的"大脑"就是引擎。

这个"大脑"对编程语言有什么要求？

| 需求 | 比喻 | 不满足的后果 |
|------|------|-------------|
| **精确控制子进程和 I/O** | 流水线上每个工位（节点）是一个独立的工人（子进程）。引擎要能启动工人、给工人材料（stdin）、收工人的成品（stdout）、工人做事太久时要能叫他停下（超时强杀） | 工人启动了关不掉；材料给了但工人没收到；工人做完了但引擎不知道 |
| **高效并发处理大量节点** | 流水线上可能有 50 个工人同时在做事。引擎不能因为等一个工人就停下整个流水线 | 一个工人卡住，所有工人都等他，流水线停转 |
| **没有 GC 停顿** | GC 是"自动垃圾回收"——编程语言每隔一段时间自动检查哪些内存不再用了，然后回收。检查期间程序暂停。在流水线比喻里，相当于每隔几分钟流水线突然停下 10 毫秒，等管理员清完垃圾再继续 | 引擎在处理高频小任务时，每几秒卡一下，总耗时比实际执行时间还长 |
| **编译器能发现错误路径** | 编译器是把人类写的代码翻译成计算机指令的工具。好的编译器能在翻译阶段（编译时）就发现你可能出错的地方，而不是等你运行时再崩溃 | 线上运行到某个特殊场景才崩溃，比编译时报错难排查 100 倍 |
| **启动快、部署简单** | 用户下载一个文件就能跑，不需要先装一堆环境。启动命令到出结果只要几十毫秒 | 用户要先装 JDK（几百 MB）、配环境变量，然后等半分钟启动 |

带着这些需求去看各个语言。

## 二、主流语言对比

### Go

**Go** 是 Google 开发的语言。它的特色是 goroutine——一种轻量级的"微型线程"（线程可以理解为一个工人，goroutine 是一个可以快速切换的超轻工人）。

**Nexus 如果用 Go：**

- **goroutine 有开销。** 每个 goroutine 至少占用 ~2KB 内存。1000 个节点并发是 ~2MB——这不多。但更大的问题是：Go 有 GC（垃圾回收）。EdgeState 和 NodeState（节点的运行时状态）是大量的小对象（几个数字、一个集合），GC 每次扫描它们都会产生停顿——流水线每隔一会儿就卡一下，因为管理员在清扫地面。

- **没有 enum + match。** enum 是"枚举类型"——限定一个值只能取几种可能性。比如"事件类型"只能是 Complete/Failed/Timeout 之一。match 是根据枚举的值来对应处理的分支结构。Go 没有这个能力，处理不同事件类型只能用 `if-else`。如果你加了一种新的事件类型，Go 编译器不会告诉你哪个 `if-else` 忘记处理它——你自己去翻代码。

- **子进程管理麻烦。** Go 的标准库提供了管理子进程的能力，但超时处理——超过指定时间要强制结束进程——需要你自己包装定时器、处理并发、手动处理竞态（两个东西同时发生导致的冲突）。Rust 的 tokio 库一句话搞定。

### Java / C#

**Java** 和 **C#** 是两种企业级语言，运行在虚拟机（JVM / .NET）上。

- **启动慢。** 虚拟机需要先启动（加载几百个类、做即时编译预热）。运行 `nexus run workflow.json` 先等 300-500ms。就像流水线启动后，管理员先花半分钟泡杯茶才开始干活。

- **GC 停顿。** Java/C# 也有 GC（垃圾回收）。当引擎每秒处理几百个节点完成事件、产生大量临时对象时，GC 会频繁触发"Stop-The-World"（暂停所有程序）来回收这些对象。流水线又要每隔几秒停下清垃圾。

- **用户要先装运行时。** 用户要跑一个 Java 程序，必须先把 JVM（几百 MB）装好，配好环境变量。C# 要装 .NET Runtime。Rust 编译成单一二进制文件（一个 .exe 或 一个文件），下载就能跑。

### Python

**Python** 是最流行的脚本语言，语法简单。

- **GIL（全局解释器锁）。** 这是 Python 的设计限制：**同一个时刻只能有一个 CPU 核心在工作。** 你开了 10 个工人在流水线上干活，其实 9 个在等那 1 个做完。对于多节点并发的编排场景来说很痛苦。

- **动态类型。** 在 Python 中，你写一个函数接受"事件类型"参数——你只能规定它是一个字符串。调用者传 "complete"、"Completed" 还是 "COMPLETE"？只有运行时才知道。Rust 的 enum 会在编译时就告诉你"你传了个不存在的值"。

- **子进程性能差。** 调用子进程后等待它完成的方法是**阻塞**的——调了等在那，不能做别的事。50 个子进程你要做 50 次等待。Rust 的异步机制可以同时等待所有子进程，不阻塞。

### Node.js / TypeScript

**Node.js** 是后端 JavaScript 运行时。**TypeScript** 是给 JavaScript 加类型的扩展。

- **单线程事件循环。** Node 的事件循环（不断收事件、处理事件、发新事件的循环）和 Nexus 的概念上匹配。但 Node 只能用一个 CPU 核心。如果某个节点需要做大量计算（比如正则匹配 10MB 文本），事件循环就被卡住了——其他节点的完成事件收不到。

- **npm 依赖地狱。** npm 是 Node 的包管理工具。一个项目依赖几十个包，每个包又依赖更多包，最后依赖树可能有几百个包。随便哪个包升级都可能出问题。

- **无内存安全。** TypeScript 可以检查类型（告诉你可以传什么值），但不能检查内存安全（不会数组越界、不会访问已经释放的内存）。Node 底层是 C++ 写的插件，这些插件经常出内存问题。

### C++

**C++** 是性能之王，几乎所有游戏引擎、浏览器、操作系统都是用 C++ 写的。

- **人类无法保证内存安全。** C++ 能做到和 Rust 一样的性能，但代价是程序员必须自己管理 everything——"这个对象什么时候销毁？""还有没有人在用它的内存？"C++ 中你忘了释放内存或者访问了已经释放的内存，程序就崩溃。在 Nexus 这样错误路径极多的系统中（节点超时、spawn 失败、配置错误），任何一个路径漏了处理都会导致内存错误。

- **没有标准的构建系统。** Rust 的 cargo 是一个命令搞定创建项目、下载依赖、编译、测试、生成文档。C++ 有 CMake / Conan / vcpkg 等一堆工具，组合起来极其复杂。

## 三、Rust 给出的答案

### 3.1 精确控制子进程

**tokio** 是 Rust 的异步运行时库。它提供了一整套管理子进程的工具：

```
想象你在等外卖：
  其他语言：你要自己定闹钟（设置定时器）、每隔一会儿去门口看看（轮询）、超时了去催外卖员（发信号）
  Rust：你告诉 tokio"帮我等这个外卖，30 分钟后没到就标记超时"
         tokio 在这期间帮你做其他事（等别的外卖），30 分钟一到自动提醒你
```

代码示例中：

```rust
let mut child = Command::new("python")     // 告诉系统：启动 python
    .stdin(Stdio::piped())                 // 我要给它传数据
    .stdout(Stdio::piped())                // 我要收它的结果
    .spawn()?;                             // 启动！spawn = "生产一个新的进程"

match tokio::time::timeout(               // 设置超时 30 秒
    Duration::from_secs(30),               // 30 秒
    child.wait(),                          // 等它结束
).await {                                  // await = "异步等待"
    Ok(Ok(status)) => { /* 正常退出 */ }
    Ok(Err(e)) => { /* 启动失败 */ }
    Err(_) => { /* 超时了，强制结束 */ child.kill().await?; }
}
```

```?``` 是 Rust 的问号操作符：如果前面的表达式出错（Err），就提前返回错误。你不用写 `if error != null { return error; }`。

> **在其他语言里，超时处理至少需要 20 行手动管理定时器 + 信号 + 竞态处理。Rust 用 `tokio::time::timeout` 一个函数搞定。**

### 3.2 无 GC 的并发

**GC（Garbage Collection，垃圾回收）**：程序运行时，会不停地产生和销毁数据。GC 就是编程语言自动帮你清理不再使用的数据。但清理的时候程序会暂停——就像扫地机器人工作时你要站着别动。

**Rust 没有 GC。** 它通过所有权（ownership）系统在编译时就确定了每个数据什么时候应该被销毁——不需要运行时暂停。

```
类比：
  Go/Java 的 GC → 扫地机器人，每过一段时间自动启动
                   "我要扫地了，你别动"
  Rust           → 家里没有扫地机器人，但有完美的收纳系统
                   每个人用完东西自动放回原位，不需要大扫除
```

**栈（Stack）** vs **堆（Heap）**：栈是程序执行时自带的一块临时存储区——函数调用时放进去，函数返回时自动清空。堆是另一块区域——需要程序员手动管理分配和释放。Rust 能把大部分数据分配在栈上，不需要堆分配。

**1000 个节点的状态（EdgeState、NodeState）只占 ~200KB 内存**——没有 GC 扫描、没有暂停、没有延迟抖动。

### 3.3 enum + match 强迫处理所有分支

**enum（枚举）**：限定一个值只能取几种可能性。

```rust
enum TrafficLight {
    Red,
    Yellow,
    Green,
}
// 一个红绿灯只能是 红/黄/绿 之一，你不能把它设成 "Blue"
```

**match（匹配）**：根据枚举的值执行对应的代码。

```rust
match light {
    TrafficLight::Red => stop(),
    TrafficLight::Yellow => slow_down(),
    TrafficLight::Green => go(),
}
```

Rust 的 match **要求你覆盖所有可能的情况**。如果你加了一个 `TrafficLight::Flashing`，编译器会报错：`pattern 'Flashing' not covered`（你没有处理闪灯的情况）。

```
对比：
  Go 的 switch：忘了写一个 case → 编译器不报错 → 运行时默默跳过
  Python 的 if-elif：忘了写一个分支 → 运行时走 else（如果没写 else 就什么都不做）
  Rust 的 match：忘了写一个分支 → 编译不通过 → 你必须处理
```

在 Nexus 中，EventType 有 Complete / Failed / Timeout 三种。如果未来加了第四种 `Cancelled`：

```rust
match event {
    EventType::Complete => handle_complete(),
    EventType::Failed => handle_failed(),
    EventType::Timeout => handle_timeout(),
    // 编译器：请告诉我 Cancelled 怎么处理
}
```

**Go 中不会有这个提示。** 你需要自己去搜索所有 switch 语句，看看有没有遗漏。

### 3.4 所有权——编译期消除竞态

**所有权（Ownership）**：Rust 中每个数据只有一个"所有者"。数据要么只读共享（多个地方可以同时读），要么独占可写（只有一个人能修改），不能同时存在。

**竞态（Race Condition）**：两个操作同时发生，互相干扰，导致结果不可预测。就像两个人同时往记分板上写分数据——最后分数乱了。

```
类比：
  GraphDef（规则书）→ 只读，所有人都可以同时查阅
  RuntimeState（记分板）→ 只有记分员能修改，其他人不能一边看一边改

  其他语言：编译器不强制这个规则——全靠程序员自觉。
           "我保证不会有人一边读记分板一边改"——但有人忘了。
  Rust：编译器强制这个规则——你不遵守编译不过。
        你不能同时"读规则书"和"改写记分板"——编译器检查你的代码是否违反了这条规则
```

在代码层面：

```rust
pub fn handle_event(&mut self, ...) {
    let edges = &self.graph.edges[edge_idx];  // 规则书——只读
    let state = &mut self.state.edge_states[edge_idx];  // 记分板——独占可改
    // 编译器保证：在 let state 的独占借用期间，没有人能通过其他引用改它
}
```

**& 和 &mut**：`&` 表示"借用只读"（你可以看，但不能改）；`&mut` 表示"借用可写"（你可以看和改，但同时期不允许其他人看）。

### 3.5 为 CLI 工具而生

**CLI（Command Line Interface，命令行界面）**——在终端敲命令跑的程序。`nexus run workflow.json` 就是一个 CLI 命令。

| 特性 | Rust | 其他语言 |
|------|------|---------|
| **启动时间** | ~5 毫秒（瞬间启动） | Java: ~500 毫秒（半秒） |
| **部署** | 一个文件（静态二进制，双击或复制就能跑） | Python：用户要装 Python 解释器 + `pip install` |
| **静态链接** | 所有依赖打到一个文件里 | 大部分语言需要依赖库文件（.dll / .so） |
| **交叉编译** | 在一台电脑上编译出其他系统/CPU 架构的程序 | 大多数语言需要目标环境的工具链 |

**静态二进制**：Rust 程序编译后会生成一个文件，包含了程序本身和所有它需要的外部库。这个文件你可以随便复制到别的电脑上跑，不用装任何东西。而 Python 程序需要一个 Python 解释器（几十 MB）才能运行。Java 程序需要 JVM（几百 MB）才能运行。

## 四、Rust 的风险和代价

### 4.1 学习曲线

Rust 的**借用检查器**（borrow checker）和**生命周期**（lifetime）对新开发者是真实的阻力——你写的代码在草稿阶段就被编译器拒绝，因为某些引用规则不满足。

```
借用检查器：编译器检查你对数据的引用（借用）是否安全的机制。
生命周期：编译器追踪一个引用有效的时间范围——"这个引用在什么范围内是可以安全使用的？"

示例（初学者常遇到）：
  let edges = self.graph.edges();    // 我要读规则书
  let state = &mut self.state;       // 我要改写记分板
  edges[0].do_something(&mut state); // ❌ 编译器：你同时做了这两件事
```

**缓解**：Nexus 的 Spec/Status 分离（规则书 vs 记分板）天然适配 Rust 的所有权模型。大部分借用问题在架构设计层面已经避免了，不是编码时才"发现"的。

### 4.2 编译时间

**编译**：把 Rust 代码翻译成机器指令的过程。因为 Rust 要做大量的编译期检查（类型检查、借用检查、泛型展开），编译时间比 Go 和 Java 长。

首次构建可能需要 2-3 分钟。**增量编译**（只编译修改过的部分）通常 <10 秒。

> **缓解**：把核心引擎（nexus-engine）和 CLI 入口（nexus-cli）分成两个包，改 CLI 不用重编引擎。

### 4.3 异步 trait 的开销

**trait**：Rust 中的"接口"概念——定义一组方法，让不同类型各自实现。

**async fn**：异步函数——调用后不立即返回结果，而是返回一个"将来会出结果"的对象（Future）。

```rust
trait NodeShell {
    async fn run(&self);  // 一个异步方法
}
```

`#[async_trait]` 是一个工具，把 async fn 转换成返回 `Pin<Box<dyn Future>>`——每次调用会产生一次**堆分配**（在堆上分配一小块内存）。

Nexus 采用**枚举分发**（enum dispatch）替代 trait 对象——避开了这个开销。在 enum dispatch 中，方法调用是直接在结构体上的，不需要间接跳转。

## 五、结论

Rust 的**三个核心特性**直接对应了 Nexus 的**三个核心架构决策**：

| Nexus 的架构决策 | 对应 Rust 的特性 |
|---------|-----------|
| **Spec/Status 分离**——GraphDef（规则书）只读，RuntimeState（记分板）独占可改 | **所有权 + 借用检查器**——编译器强制区分只读和独占可写 |
| **事件驱动 + 三种事件类型对称处理**——EventType 是 Complete/Failed/Timeout 的枚举 | **enum + match**——编译器强迫你处理所有事件类型，新增类型时提示哪里遗漏了 |
| **单线程事件循环 + 高并发**——一个线程管理所有事件，通过 tokio 异步等待子进程 | **async/await + tokio**——用同步的写法写异步逻辑，没有多线程锁竞争 |

**其他语言也能实现同样的架构——但需要更多的手工保证（纪律）和更多的代码（处理 GC、错误路径、并发模型）。Rust 的特性把这些架构决策从"运行时约定"变成了"编译期保证"。**

> Rust 不能让你写得更快，但它让你写的每个运行时路径都在编译期被检查过了。
