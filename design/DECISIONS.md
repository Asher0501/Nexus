## 一、语言选型：为什么选择 Rust

Nexus 做的事情：**启动各种程序（节点）、等它们做完、根据结果决定下一步、同时管理很多个节点的运行。**

这个"大脑"对编程语言有什么要求？

| 需求 | 比喻 | 不满足的后果 |
| --- | --- | --- |
| **精确控制子进程和 I/O** | 流水线上每个工位（节点）是一个独立的工人（子进程）。引擎要能启动工人、给工人材料（stdin）、收工人的成品（stdout）、工人做事太久时要能叫他停下（超时强杀） | 工人启动了关不掉；材料给了但工人没收到；工人做完了但引擎不知道 |
| **高效并发处理大量节点** | 流水线上可能有 50 个工人同时在做事。引擎不能因为等一个工人就停下整个流水线 | 一个工人卡住，所有工人都等他，流水线停转 |
| **没有 GC 停顿** | GC 是"自动垃圾回收"——编程语言每隔一段时间自动检查哪些内存不再用了，然后回收。检查期间程序暂停。在流水线比喻里，相当于每隔几分钟流水线突然停下 10 毫秒，等管理员清完垃圾再继续 | 引擎在处理高频小任务时，每几秒卡一下，总耗时比实际执行时间还长 |
| **编译器能发现错误路径** | 编译器是把人类写的代码翻译成计算机指令的工具。好的编译器能在翻译阶段（编译时）就发现你可能出错的地方，而不是等你运行时再崩溃 | 线上运行到某个特殊场景才崩溃，比编译时报错难排查 100 倍 |
| **启动快、部署简单** | 用户下载一个文件就能跑，不需要先装一堆环境。启动命令到出结果只要几十毫秒 | 用户要先装 JDK（几百 MB）、配环境变量，然后等半分钟启动 |

### 主流语言对比

#### Go

**Go** 是 Google 开发的语言。它的特色是 goroutine——一种轻量级的"微型线程"。

**Nexus 如果用 Go：**

- **goroutine 有开销。** 每个 goroutine 至少占用 ~2KB 内存。但更大的问题是：Go 有 GC。EdgeState 和 NodeState 是大量的小对象，GC 每次扫描它们都会产生停顿。
- **没有 enum + match。** Go 没有枚举匹配，处理不同事件类型只能用 `if-else`。加新类型时编译器不会提示哪里遗漏。
- **子进程管理麻烦。** 超时处理需要手动包装定时器、处理并发、处理竞态。Rust 的 tokio 一句话搞定。

#### Java / C#

**Java** 和 **C#** 运行在虚拟机（JVM / .NET）上。

- **启动慢。** 运行 `nexus run workflow.json` 先等 300-500ms。
- **GC 停顿。** 当引擎每秒处理几百个节点完成事件时，GC 频繁触发"Stop-The-World"。
- **用户要先装运行时。** Rust 编译成单一二进制文件，下载就能跑。

#### Python

**Python** 是最流行的脚本语言。

- **GIL（全局解释器锁）。** 同一个时刻只能有一个 CPU 核心在工作。对于多节点并发的编排场景来说很痛苦。
- **动态类型。** 调用者传 "complete"、"Completed" 还是 "COMPLETE"？只有运行时才知道。
- **子进程性能差。** 调用子进程后等待是阻塞的。Rust 的异步机制可以同时等待所有子进程。

#### Node.js / TypeScript

- **单线程事件循环。** 如果某个节点需要做大量计算，事件循环就被卡住了。
- **npm 依赖地狱。** 依赖树可能有几百个包。
- **无内存安全。** TypeScript 不能检查内存安全。

#### C++

**C++** 是性能之王。

- **人类无法保证内存安全。** 在 Nexus 这样错误路径极多的系统中，任何一个路径漏了处理都会导致内存错误。
- **没有标准的构建系统。** C++ 有 CMake / Conan / vcpkg 等一堆工具。

### Rust 给出的答案

#### 1. 精确控制子进程

**tokio** 是 Rust 的异步运行时库。它提供了一整套管理子进程的工具：

```
想象你在等外卖：
  其他语言：你要自己定闹钟、每隔一会儿去门口看看、超时了去催外卖员
  Rust：你告诉 tokio"帮我等这个外卖，30 分钟后没到就标记超时"
         tokio 在这期间帮你做其他事
```

```rust
let mut child = Command::new("python")
    .stdin(Stdio::piped())
    .stdout(Stdio::piped())
    .spawn()?;

match tokio::time::timeout(
    Duration::from_secs(30),
    child.wait(),
).await {
    Ok(Ok(status)) => { /* 正常退出 */ }
    Ok(Err(e)) => { /* 启动失败 */ }
    Err(_) => { /* 超时了，强制结束 */ child.kill().await?; }
}
```

> **在其他语言里，超时处理至少需要 20 行手动管理定时器 + 信号 + 竞态处理。Rust 用 `tokio::time::timeout` 一个函数搞定。**

#### 2. 无 GC 的并发

**Rust 没有 GC。** 它通过所有权（ownership）系统在编译时就确定了每个数据什么时候应该被销毁——不需要运行时暂停。

**1000 个节点的状态（EdgeState、NodeState）只占 ~200KB 内存**——没有 GC 扫描、没有暂停、没有延迟抖动。

#### 3. enum + match 强迫处理所有分支

Rust 的 match **要求你覆盖所有可能的情况**。如果加了一个新变体，编译器会报错。

```
对比：
  Go 的 switch：忘了写一个 case → 编译器不报错 → 运行时默默跳过
  Python 的 if-elif：忘了写一个分支 → 运行时走 else
  Rust 的 match：忘了写一个分支 → 编译不通过 → 你必须处理
```

#### 4. 所有权——编译期消除竞态

**所有权（Ownership）**：Rust 中每个数据只有一个"所有者"。数据要么只读共享，要么独占可写，不能同时存在。

```
类比：
  GraphDef（规则书）→ 只读，所有人都可以同时查阅
  RuntimeState（记分板）→ 只有记分员能修改，其他人不能一边看一边改
```

#### 5. 为 CLI 工具而生

| 特性 | Rust | 其他语言 |
| --- | --- | --- |
| **启动时间** | ~5 毫秒（瞬间启动） | Java: ~500 毫秒 |
| **部署** | 一个文件（静态二进制） | Python：用户要装解释器 + `pip install` |
| **静态链接** | 所有依赖打到一个文件里 | 大部分语言需要依赖库文件 |
| **交叉编译** | 在一台电脑上编译出其他系统/CPU 的程序 | 大多数语言需要目标环境的工具链 |

### Rust 的风险和代价

#### 学习曲线

Rust 的**借用检查器**（borrow checker）和**生命周期**（lifetime）对新手是真实的阻力。

**缓解**：Nexus 的 Spec/Status 分离（规则书 vs 记分板）天然适配 Rust 的所有权模型。

#### 编译时间

首次构建可能需要 2-3 分钟。**增量编译**（只编译修改过的部分）通常 <10 秒。

> **缓解**：把核心引擎（nexus-engine）和 CLI 入口（nexus-cli）分成两个包，改 CLI 不用重编引擎。

#### 异步 trait 的开销

Nexus 采用**枚举分发**（enum dispatch）替代 trait 对象——避开了 `#[async_trait]` 的堆分配开销。

### 结论

| Nexus 的架构决策 | 对应 Rust 的特性 |
| --- | --- |
| **Spec/Status 分离**——GraphDef 只读，RuntimeState 独占可改 | **所有权 + 借用检查器**——编译器强制区分只读和独占可写 |
| **事件驱动 + 三种事件类型对称处理**——EventType 是 Complete/Failed/Timeout 的枚举 | **enum + match**——编译器强迫你处理所有事件类型，新增类型时提示哪里遗漏 |
| **单线程事件循环 + 高并发**——一个线程管理所有事件，通过 tokio 异步等待子进程 | **async/await + tokio**——用同步的写法写异步逻辑，没有多线程锁竞争 |

**其他语言也能实现同样的架构——但需要更多的手工保证（纪律）和更多的代码（处理 GC、错误路径、并发模型）。Rust 的特性把这些架构决策从"运行时约定"变成了"编译期保证"。**
